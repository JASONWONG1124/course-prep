Rules System · CH 06

CLAUDE.md 与规则系统

理解入口文件、规则、Source of Truth 和项目执行边界。

知识小节

CLAUDE.md：项目入口文件

学习目标

这一节用于理解「CLAUDE.md 与规则系统」里的关键概念、使用场景和判断边界。

知识讲解

不同项目的资料、结构和任务类型会有差异，但都会需要一个共同的入口文件来告诉 Agent 如何理解项目。

在 Claude Code 项目中，每个项目文件夹都可以放置一个名为 CLAUDE.md 的入口文件。这个文件使用全大写名称，用来声明项目规则、工作方式和 Agent 进入项目后应该优先理解的内容。

这个文件很特殊

它不是给人看的记录，也不是普通的项目文件

它是一个规则文件

每次你启动 Claude Code，它做的第一件事就是读这个文件 CLAUDE.md 决定了 Claude Code 进入这个项目时，

第一眼看到的世界是什么样的 读完之后，它就知道了：你是谁，你在做什么，你希望它怎么帮你 从那一刻起，它就不是一个对你一无所知的陌生 AI 了，它是一个已经了解你的情况、知道你的偏好、准备好开始干活的协作者

不是「可能会读」，不是「如果它心情好会读」，是一定读，每次都读，没有例外

一个字母的差别，决定了它是否真的会读

这里分成两种用法，一个叫全局应用，一个叫项目应用 放置的位置不同，影响的范围也不同 全局应用是放在 claude code 的根目录「大本营」下 项目应用就是放在对应项目下

知识小节

不同项目的 CLAUDE.md 该写不同规则

学习目标

这一节用于理解「CLAUDE.md 与规则系统」里的关键概念、使用场景和判断边界。

知识讲解

CLAUDE.md 没有标准答案 不同的项目，CLAUDE.md 长得完全不一样

用来写代码的项目，CLAUDE.md 长这样

代码项目里全是技术规范 代码的世界里没有风格偏好，没有人设，没有表达调性。代码对就是对，错就是错。

所以做开发项目的 CLAUDE.md 里，放的是技术信息：

这个项目用的是什么技术栈——前端用 React，后端用 Python，数据库用 PostgreSQL。 项目的架构是什么——文件夹怎么分，每个模块负责什么，代码规范是什么。 项目背景是什么——这个产品要解决什么问题，现在做到哪个阶段，这次要实现什么功能。

AI 读完这些，它就知道自己在一个什么样的技术环境里工作，不会给你推荐用不了的技术，不会写出风格不一致的代码

个人知识库里是偏好和架构说明 因为是自用的

自媒体项目里是内容规则

知识小节

全局规则和项目规则要分开

学习目标

这一节用于理解「CLAUDE.md 与规则系统」里的关键概念、使用场景和判断边界。

知识讲解

CLAUDE.md 可以分成全局规则和项目规则两类理解。

全局 CLAUDE.md，应该放个人偏好、协作习惯、表达审美、判断方式。 因为它是跟着你这个人走的。

你不管打开哪个项目，你都希望 Agent 知道：

你讨厌 AI 味。

你喜欢先抓核心矛盾。

你重视第一性原理、机制拆解和结果验证。

你希望它不要上来就执行，而是先判断问题是否成立。

你希望中文表达有人的判断痕迹，不要模板化、工业化、翻译腔。 这些东西不是某一个项目的规则，而是你这个人的工作方式。

所以它应该放在全局 CLAUDE.md 里。

项目级 CLAUDE.md 解决的是另一件事。 项目级 CLAUDE.md 不是用来描述你是谁，而是用来告诉 Agent：

它现在进的是哪个项目。

这个项目服务什么工作。 项目里有哪些关键目录。 哪些文件是入口文件。 哪些资料是源材料。 哪些文件不能乱改。 遇到不同任务时，应该先读哪里。 资料之间是什么关系。

全局 CLAUDE.md 塑造协作人格。 项目 CLAUDE.md 建立项目入口。

一个解决「你是谁」。

一个解决「这个项目是什么」。

知识小节

CLAUDE.md 从哪里来，怎么持续更新

学习目标

这几页讲 CLAUDE.md 的生成和维护：初始规则可以让 Agent 读项目、反问你、替你写；后续每一次摩擦都应该被总结成规则写回入口文件。

知识讲解

CLAUDE.md 里面的内容怎么来的 方式有很多，如果你项目里的资料已经全部堆放完成了，就可以用 Claude Code 全面读一下项目里的所有内容，让它自己去感受：

里面有什么内容；

里面的架构是什么；

你是做什么的，是一个什么样的人；

你的业务流程是怎么样的。 最后，让它给你写出一版 CLAUDE.md

又或者课程中把想象力再打开一下，我们就可以跟 Claude Code 说： 「接下来你可以反问我 10~15 个问题，来了解清楚我到底是一个什么样的人，我有什么偏好，我喜欢什么样的工作风格？最后你再给我定制一个专属于我的 CLAUDE.md。」 不需要我们去一行一行地写，因为很多人往往总结不到自己的，也没办法彻底认清自己

所以这个方式好的地方就是引入一个第三方，让 Claude Code 来问我们问题，在里面感受到我们，把我们这个人格给塑造出来。学员会发现，其实 Claude Code 往往会比我们更我们自己

使用技巧：

我们在使用 Claude Code 的过程中，会发现很多跟他的一点摩擦，他做得不好的地方，或者想让他以后不要再犯的地方。例如，你可能是一个表达需求不太清晰的人，一时半会没有办法改变自己的表达习惯。 那你在跟 Claude Code 对话的过程中会发现，它经常没有深度地理解你的话就开始执行，带着一个很模糊、没有判定边界的方向去做，最后出来的结果不是你想要的。 这时，你就可以明确地跟它说，我平时的表述可能不太清楚，但是你每一次都带着我一些模糊不清的表述就开始干活，没有跟我多次澄清我的问题，没有把一个任务的边界和它的细节全部定死，就导致很多时候，你给我做的东西是不对的。

以后我不想出现这个情况，你把需要反省的这些地方，编写好一些规则，然后放到 CLAUDE.md 文件里面去。

学员合作越久，CLAUDE.md 越懂你 每一次摩擦都是一次校准，每一条新规则都是一次进化

可复用材料

prompt

CLAUDE.md 生成提示词：先让 Agent 反问，再写规则

CLAUDE.md 不适合凭空写。更稳的做法是先让 Agent 反问 10-15 个问题，把项目目标、文件边界、交付标准和摩擦点问清楚，再让它生成入口规则。

我想为这个项目生成一份 CLAUDE.md / AGENTS.md。
请你不要直接开始写，先反问我 10-15 个关键问题。

问题要覆盖：
- 这个项目到底服务什么业务目标
- 项目里有哪些关键资料，分别放在哪里
- 哪些文件可以读，哪些文件不能改
- 产出应该放到哪里，命名规则是什么
- 我希望你优先遵守哪些表达风格、判断标准和禁区
- 过去协作中最容易出现的摩擦点是什么
- 什么情况下你必须先问我，不能擅自行动

等我回答完以后，再帮我生成一份短而清楚的 CLAUDE.md / AGENTS.md。
规则只写真正会影响结果的内容，不要写空泛原则。

知识小节

Source of Truth：让 Agent 知道去哪里找

学习目标

这几页讲 Source of Truth：当项目资料变多，问题不是有没有资料，而是 Agent 能不能找到、该先信谁、入口文件有没有告诉它索引存在。

知识讲解

它是一个独立的文件，被 CLAUDE.md 单独引用的

它的作用是告诉 Agent： - 项目里有哪些重要文件。 - 每类资料大概在哪里。 - 每个文件夹是做什么的。 - 某类任务应该优先读哪些文件。 - 目标文件和相关文件之间有什么关系。 当用户要求找资料时，Agent 应该从哪里开始找。

`Source of Truth` 是让 Agent 在大规模本地资料里快速定位知识的权威入口

为什么要建立数据权威索引 当本地资料膨胀到几百万字、上千万字以后，模型不可能每次都完整读取

> 索引不能只是单独存在，它必须被写进 `AGENTS.md` 和 `CLAUDE.md`。

原因很简单：

Agent 启动时通常会读取入口文件 - 但它不会自动知道项目里有一个 Source of Truth - 如果入口文件没有提到索引，Agent 就不知道要用它 这样即使建立了索引，也等于没建立

而且索引还有个好处，文件名通常是承载不了很多观点、案例和方法的，但是在索引里，一个文件，你可以加描述，加标签，真实用途

所以索引应该怎么建呢？ 需要人写不？ 让 AI 写

可以这儿跟 AI 说 反正表达的，就是那个意思就行了

复盘材料

可复用材料

prompt

Source of Truth 建立提示词

当项目资料变多以后，Source of Truth 不是把所有知识再复制一遍，而是建立权威索引。它解决「资料在哪里、任务该先读哪里、冲突时信谁」的问题。

当前项目文字量很大，超过单次模型上下文，不能靠每次全量读取资料来工作。

请为这个项目建立一份 Source of Truth（数据权威索引），用于让 Agent 快速理解项目资料结构、关键文件用途、任务到文件的读取路径。

请你完成：
1. 扫描项目目录，识别核心资料、规则文件、产出文件和原始材料。
2. 为每类文件写清楚：它回答什么问题，什么时候应该读取。
3. 建立「任务类型 → 必读文件 / 参考文件」的读取路径。
4. 标注冲突处理规则：如果多个文件说法不一致，应该优先相信哪一类文件。
5. 写清维护规则：新增、删除、移动资料后，索引应该如何更新。

建立完成后，请把该索引文件的路径写入 `AGENTS.md` 和 `CLAUDE.md`，让不同 Agent 启动项目时知道这个索引存在。

索引不要被设置为每次必读，只在需要找资料时读取。
workflow

测试 Source of Truth 有没有用

索引写完以后，不要只看它格式漂不漂亮，要用任务反测。

  1. 找一个真实任务，比如「根据过往案例写一条新选题」。
  2. 要求 Agent 先只读 Source of Truth，不读全量资料。
  3. 看它能不能判断接下来应该打开哪些文件。
  4. 如果它找错文件，说明索引路径不清楚，要回去改索引。
  5. 如果它每次都想读全部文件，说明索引边界写得太重，要改成按需读取。

知识小节

动手之前先立边界

学习目标

这一节用于理解「CLAUDE.md 与规则系统」里的关键概念、使用场景和判断边界。

知识讲解

开始搭建前，需要先判断项目已有资料的积累程度。

开始操作前，应确认 VS Code、Claude Code 和模型 API 已经可用。

文件夹结构，就是业务结构的映射 项目结构是业务关系的结果

如果你不把你的整个业务，从你起床，到你赚到钱的那一刻，中间所有的体系完全展开，是没办法搭一个和你天然适配的文件夹架构的

但是绝大多数人，是很难完全通过表达和口述，把自己整个复杂的业务体系 & 地图给摊开的

你的表达，同样充满了歧义和不确定性 想不明白 和 讲不清楚，对 AI 来说，就是灾难

在建文件夹之前，先把自己的业务说清楚。不能带着一个模糊的说辞，直接要求 AI 生成项目

大多数人对自己的业务有直觉但没有语言。他们知道自己在做什么、讨厌什么、擅长什么，但从来没有把这些东西**写成一份清晰的文档**。这个 skill 做的就是帮他们把散落在脑子里的碎片，整理成一份有结构、有温度、有灵魂的自我画像。

自我业务画像的价值不只是「给 AI 看」——它本身就是一次自我认知的梳理。做完之后，「我从来没这么清楚地看过自己在做的事」。这是附带价值，但很重要。

已有资料较充分时，可以先让 AI 读取原始材料，从中提取业务结构、内容资产、表达风格和关键判断，再生成初始业务地图。

资料积累不足时，需要先通过问答方式把业务目标、受众、交付方式和真实限制说清楚。这个方法的效果取决于表达者能否准确描述自己的业务。
