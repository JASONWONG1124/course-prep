Hands-on Project · CH 07

动手搭建个人项目基础

从业务流程出发，搭建个人 AI 项目的初始地基。

知识小节

SANHESkill：先把人的业务说清楚

学习目标

这几页进入实操：搭文件夹之前先理清业务，world-mapper 负责把业务、人设和资产写成自我画像，world-builder 再把它翻译成项目结构。

知识讲解

课程提供的 SANHESkill 用于把业务、人设、资产和项目结构整理成 Agent 可以长期调用的材料。这里先理解它们在项目搭建链路中的位置，具体原理可以放到后续章节展开。

其中

其中 World-mapper 这个 skill，就是给学员产出一份完整的「业务 + 人设 + 资产」自我画像。 自我画像不是问卷结果，不是标签集合，它是一份让下一个 Claude Code 读完后能「10 秒之内成为你」的文件，同时也是自己看完会觉得「被精准描述」的镜子

它是让 AI 成为你的关键

world-builder 读取自我画像后，会继续追问 3-5 个和系统用途有关的问题，用来确认 AI 需要进入哪些工作流、管理到什么粒度，以及哪些任务应该优先进入项目。

你希望 AI 帮你做哪几件具体的事？哪些工作流要进系统？数据想管到什么粒度？

架构不是一次成型的，是自己根据实际情况来做成自己最舒服，最贴合的状态

前两步做完了，你有了一个空的文件夹架构。 这一步要做的事情只有一件：把你现有的所有业务资料，按照文件夹架构的分类，全部放进对应的位置。

但在放之前，有一个动作不能跳过——资料清洗。

你现在手里的资料，可能散落在各种地方，各种格式。 微信聊天记录、Word 文档、PDF、截图、Excel 表格、语音备忘录、随手记的便利贴——这些东西格式混乱，AI 读起来效率低，理解质量差。 放进去之前，必须先统一格式。

资料清洗不需要一次做完

你现在有什么就先放什么，哪怕只有三份文件，也比空文件夹强。 知识库是生长的，不是一次性建好的。先把现有的整理好放进去，往后每产出一份新内容、每沉淀一个新判断，就同步更新进来。 文件夹跟着你的业务一起长，它永远不会有「完成」的那一天。

课堂实操的重点不是让大家记住几个 skill 名字，而是理解它们的顺序：先把人看清楚，再搭项目骨架，再把资料放进去，最后给文件之间织网。world-anchor 和 world-depth 是延伸的认知工具，用来处理定位和目标拆解。

知识小节

资料清洗、放置和维护原则

学习目标

这几页讲资料进入系统之后怎么处理：先清洗成 AI 友好的格式，再让 Agent 按架构放置，后续维护坚持人判断、AI 执行。

知识讲解

第一种：手动放 就是你自己判断每份资料属于哪个文件夹，自己一个个拖进去。

这个方式可以用，但不太建议。 原因很简单——你的文件可能有几十份甚至上百份，一个个判断、一个个移动，费时费力。更重要的是，人在做这种重复性工作的时候很容易放错位置，或者遇到一份资料不知道该放哪里就随手扔进去，久了文件夹就乱了。 不要用你最宝贵的时间做最低价值的事情

第二种：用 Agent 来放 把所有清洗好的资料扔进一个临时文件夹或者我们已经建好的那个文件夹，然后打开 Claude Code，告诉它： ▎ 「这个文件夹里有我所有整理好的资料，请你读取每一份文件的内容，根据我的项目架构，判断它应该放在哪个位置，然后帮我自动分类放置好。」

它会自己读每一份文件，理解内容，对照你的项目架构，把每份资料放进最合适的文件夹。 放完之后它会给你一份报告——哪些文件放了哪里，哪些文件它不确定归属、需要你来判断。

这个方式好在哪里？

第一，它比你更仔细。它不会累，不会随手乱放，每份文件它都认真读过再判断。

第二，它不确定的会告诉你。遇到一份文件确实跨类别、难以归属，它不会乱猜，它会列出来问你，最终决策权还在你手里。

第三，它顺手帮你发现问题。在读资料的过程中，它可能会告诉你：「你有几份文件内容重复，建议合并」，或者「你的数据文件格式不统一，建议统一一下」。 这些是你自己手动放的时候不会发现的东西。

例如，拿个案例，很多零散的文件，已经分类好了，还要有一个」

你的灵魂 你的骨架 资料填入

知识库的维护：人看，AI 动 文件夹搭好了，资料放进去了。 接下来有一个很多人没想清楚的问题——这个知识库以后应该怎么维护？谁来维护？ 答案很反直觉：人不应该直接碰这个文件夹，甚至不应该增删改，你只有查看的权利

为什么？ 所有文件不是孤立的，删掉文件后，索引里可能还保留说明；下次 Agent 按索引找，就会出现「目录说有，架上没有」

你自己动了，Agent 是不知道的，那索引里面还有，久而久之，整个对应关系就会乱，整个知识库就会开始腐烂了，哪哪都不对了

创建、删除、决定文件放哪里不应该由人做。因为文件夹复杂以后，人根本不知道新文件应该放在哪，甚至不知道是否需要新建分类。 文件由 Agent 管理，由 Agent 编辑，由 Agent 维护；自己只和 Agent 沟通

这是大多数人知识库最终的结局——起了个头，没有坚持下去

一旦知识库进入上千、上万文件规模，文件就不再是一个个独立文档，而变成一张关系网。

一个文件名变了，可能影响索引；一个文件删了，可能影响 Source of Truth；

一个文件放错目录，Agent 以后就会按错误路径调用资料。人手动改，很容易只改眼前一个点，不会维护整张网

人删文件，只会删文件；Agent 删文件，它还会很聪明的同时删索引，删掉对应的链接，完全抹掉 人创建文件，只会新建一个文档；Agent 创建文件，会把它放到合适位置，并给索引新增记录

workflow

操作路径：你的灵魂 → 你的骨架 → 资料填入 → 织网

这张图必须保留，因为它就是学员回去跑 SANHESkill 的顺序。

  1. 你的灵魂：跑 world-mapper，通过深度对话产出 自我画像.md。
  2. 你的骨架：跑 world-builder，基于自我画像生成 CLAUDE.md、SOURCE_OF_TRUTH.md、人设锚点和项目结构。
  3. 资料填入：把过往作品、案例、复盘、客户资料、课程资料按项目结构放进去。
  4. 织网：当 Markdown 文件变多以后，跑 world-weaver，补 YAML frontmatter 和 Obsidian 双向链接。
  5. 装肌肉：系统有真实内容之后，跑 world-skill，判断哪些工作流值得未来做成专属 Skill。
prompt

资料清洗与自动分类提示词

这段要给学员留着。真正省时间的地方，是让 Agent 按项目结构移动和归类资料，人负责判断结果是否合理。

这个文件夹里有我所有整理好的资料。

请你读取每一份文件的内容，根据我的项目架构，判断它应该放在哪个位置，然后帮我自动分类放置好。

处理要求：
- 不要只根据文件名判断，要读内容。
- 先给出分类方案，让我确认。
- 确认后再移动文件。
- 如果有文件无法判断归属，单独列出来，不要硬塞。
- 移动完成后，更新 Source of Truth 或资料索引。
- 原始资料不要删除，涉及覆盖、重命名或批量移动时先说明影响。
workflow

维护原则：人看，AI 动

文件夹搭好以后，最容易坏在维护上。人的角色不是每天手动拖文件、改文件名、删目录，而是查看现状、做判断、把变化告诉 Agent。真正的创建、移动、重命名、更新索引，应该让 Agent 执行。

  1. 人负责查看：随时打开文件夹，知道知识库现在长什么样。
  2. 人负责判断：告诉 Agent 新资料是什么、旧资料哪里变了、哪些内容需要更新。
  3. AI 负责执行：写成什么格式、放在哪个文件夹、怎么命名、要不要更新索引，都交给 Agent 处理。
  4. 不要自己随手删改文件。一个文件名变了，可能影响 Source of Truth；一个文件放错目录，Agent 以后就会沿着错误路径调用资料。
  5. 维护的核心不是勤快，而是保持文件、索引、关系网络同步。

知识小节

邀请三位同学

学习目标

这一节用于理解「动手搭建个人项目基础」里的关键概念、使用场景和判断边界。

知识讲解

业务地图和项目架构完成后，一个基础项目系统就已经出现。此时 Agent 已经拥有初始业务上下文，下一步要解决的是知识之间的关系网络。

它们已经存在了，Agent 已经有你的业务上下文了，也知道 xxxxx，也知道 xxxxxx CLAUDE.md 里面，也清晰的说清楚了索引，告诉 AI 我有什么，你要去哪里找了 可是，这么庞大的知识，只属于弱链接维持 知识库还是一堆独立的文件，每个文件各自存在，各自表达自己的内容 这时候，Obsidian 出现了 Agent 会知道需要什么文件去哪里找，每个文件里有什么，但它不知道这些文件之间有什么关系

刚才通过彻底梳理清楚了每个人自己的业务地图和专属项目架构搭建，一个基础的项目系统已经出现了 Agent 已经有了你的业务上下文，它知道你是谁，知道你在做什么赛道，知道你的表达风格，知道你的受众是谁。 CLAUDE.md 里也设计好了，索引也讲了，不过索引不太急，学员回头慢慢弄也行，索引就会告诉 AI 你有什么、要去哪里找

你的「受众画像」和你的「选题方法论」之间，可能会有深度的逻辑关联。

你的「数据规律」和你的「爆款文案」之间，可能也会有直接的因果关系。

当你的世界观和你的每一条爆款文案之间有链接，AI 就能理解为什么这条文案跑得通——不只是知道它跑通了，而是理解背后的逻辑，然后复制这个逻辑到新的内容上。 这种知识和知识之间的网络，就是双链。 而让这个双链在你的知识库里真正建立起来的工具，就是接下来要讲的——Obsidian「转」

在我们的整个课程里面，Agent 协作业务的整个链路是：Claude Code + Obsidian + 飞书

Obsidian、Claude Code 和飞书不是三套独立系统，而是同一条工作链路中的三个职责位置

前面我们已经用了很长的篇幅，讲清楚了 Agent 的所有东西，也详细地讲了关于 Claude Code 的文件夹和项目机制到底怎么用。

课后需要继续完善文件夹结构、填充知识库，并把真实业务中的资料、案例和判断持续沉淀进去。

接下来，为了解决刚才我讲的「知识链接」问题，我们就会用一个软件叫 Obsidian。
