Obsidian 本地知识库

把 Markdown 文件沉淀成可阅读、可关联、可被 Agent 使用的知识系统。

知识小节

Obsidian：打开本地 Markdown 世界

学习目标

这一节用于理解「Obsidian 本地知识库」里的关键概念、使用场景和判断边界。

知识讲解

常见疑问是：既然已经有飞书、Notion、语雀、印象笔记等工具，为什么还要使用 Obsidian。这个问题的关键不在笔记软件本身，而在于本地 Markdown 文件夹是否能被 Agent 稳定读取和维护。

这个问题问得很对。 因为如果只是「写笔记」，这些工具都能写

飞书文档甚至更好看，更适合协作，也更适合直接分享给别人。

但我们今天不是在讲普通笔记

我们从头到尾讲的都是 Claude Code 的项目知识库 这两件事的要求是不一样的

飞书、Notion 这类文档工具，本质上是云端文档工具

你的内容主要存在它们的服务器里

你打开的是一个网页、一个账号、一个在线空间

但 Claude Code 的工作方式不是这样的

Claude Code 最擅长进入你电脑本地的一个文件夹 在你的项目，在你的文件夹里面工作

Agent 读取这个文件夹里的资料，理解这个文件夹里的规则，然后在这个文件夹里生成文件、修改文件、整理结构、执行任务。

所以我们前面一直讲： 文件夹即世界

这个世界最好就在你的电脑本地 在本地，用 markdown 无限沉淀，资料才是最稳的

Obsidian 本质上不是一个「云笔记软件」。

它打开的不是一个账号，不是一个网页数据库

它打开的是你电脑上的一个本地文件夹 里面的每一篇笔记，本质上都是一个 Markdown 文件 也就是我们前面一直讲的 .md 文件

我打开你电脑里的一个文件夹，然后把这个文件夹里的 Markdown 文件组织成一个知识系统

那有人会问了： 既然它本质上就是一堆文本文件，那我用 txt 不行吗？

可以

但 txt 也能写东西

今天想了一个选题 用户为什么不愿意付费 短视频开头要制造冲突

但这些内容写完之后，基本就是一篇一篇躺在那里。你以后要靠文件名、文件夹、搜索去找，它就是散的，没有任何关联的

但 Obsidian 是统一管理的

那 Obsidian 和 Claude Code 到底是怎么联动到一起的？

Claude Code 是怎么工作的，以文件夹项目起点开始，在里面闷头干活的

那 Obsidian 也是项目制的，一个文件夹，就是一个知识库，里面会有很多业务文件

那就巧了，它们都是打开文件夹来干活的，以文件夹为中心的

所以，链接点就出现了

这是我的知识库文件夹，我用 claude code 打开它，也用 Obsidian 打开它 打开的是同一个文件夹

Claude Code 看的是文件夹。

Obsidian 打开的也是这个文件夹。

它们天然能接上

知识小节

本地知识库和 Agent 的分工

学习目标

这一节用于理解「Obsidian 本地知识库」里的关键概念、使用场景和判断边界。

知识讲解

Obsidian 和本地 Markdown 文件夹面对的是同一批文件。文件夹里的内容变化，Obsidian 中会同步显示；Obsidian 中修改文件，本地文件也会同步变化。

「要演示」

Obsidian 的价值，不只是存文字，存文件，而是让文件之间产生关系

它有文件夹结构，让资料能被分类

还有个很厉害的东西，叫双链，让一篇笔记可以连接另一篇笔记

它有反向链接，让你知道哪些内容提到过这个概念

它有搜索、标签、图谱，让一个项目里的知识不再是一堆散落的文件，而是一张逐渐长出来的关系网

所以 Obsidian 不是让你的知识变高级。

它是让你的知识变得更容易被人管理，也更容易被 AI 读取。

我打开你电脑里的一个文件夹，然后把这个文件夹里的 Markdown 文件组织成一个知识系统

它看起来像笔记软件，但它的底层逻辑不是「我帮你存笔记」 而是管理知识

Obsidian 不只是「存内容」，而是强调「连接内容」。 很多笔记软件更像资料柜： 我建一个文件夹； 我放一篇笔记； 我分门别类； 我要的时候再找。

Obsidian 更像一张知识网络： 这篇笔记和那篇笔记有关；

这个概念被哪些地方提到； 这套方法论能不能连接到某个案例；

一个选题、一个用户痛点、一个表达风格，能不能互相长出关系。

所以它特别适合做： 个人知识库

知识小节

YAML 和双链：给文件写封面和关系

学习目标

这几页说明 Markdown 文件不能只放正文，YAML 是文件封面，双向链接是文件之间的关系，二者让人和 Agent 都更容易调用知识。

知识讲解

MD 文件里直接放知识原文就可以了吗 可以，一点毛病没有，但是会有几个缺点，会让 AI 读它的效率不够高，理解不够透彻

它读到的是一坨没有被消化的原始信息 Agent 不好检索，它一看，就得把整篇东西看完，浪费上下文

我们不是让 AI 给我们干活的，我们是给 AI 服务的 服务这么差，它能把活干漂亮吗？

所以，知识 markdown 文件，里面应该要有一些固定的格式

这是一篇标准的 markdown 文件格式

你看图一顶部那个红框区域，里面有 title、claim、use、category、tags 这些字段

这个区域叫 YAML，它是每篇 Markdown 文件最顶部的一块元数据区域，用三条横线 —— 包裹起来

元数据是什么意思？ 「元」就是「关于这个东西本身的描述」。 文章的正文是内容，YAML 是关于这篇文章的描述——这篇文章叫什么、核心观点是什么、用来做什么、属于哪个分类、有哪些标签。

它不是给人读的正文，是给系统读的索引信息。

YAML 对 Agent 和 Obsidian 来说，就是这本书的封面信息

底部红框区域，里面有三条带下划线的蓝色链接，每一条链接到另一篇文章，并且附上了说明这两篇文章之间关系的一句话 这叫双向链接，写法是 [[文件名]]

普通链接是单向的——A 指向 B，但 B 不知道 A 的存在。 双向链接是双向的——A 链接了 B，B 那边会自动出现「有人链接了我」的记录。两篇文章互相知道对方的存在。 为什么这很重要？

你的知识不是孤立存在的，它们之间有真实的逻辑关联。 这篇「自媒体选题理解」，和「莱考夫哲学对选题的应用」之间有关联，和「专家型 IP 的致命错误」之间有关联，和「不会算计的内容都是噪音」之间也有关联。 这些关联在你脑子里是清楚的，但在独立的文件系统里是断掉的。 双向链接把这些断掉的关联重新连接起来。 对 AI 来说，双向链接意味着它不再只是读一篇文章，它能顺着链接走，读完这篇再读相关的那几篇，理解它们之间的逻辑关系

知识小节

YAML + 双链的提示词和 world-weaver

学习目标

这几页给出可直接复做的 YAML + 双链处理方法：可以用完整提示词，也可以用 world-weaver，把散落 Markdown 文件织成知识网络。

知识讲解

处理 YAML 和双向链接有两种方式：一种是直接使用完整提示词，让 Agent 按要求补充 YAML 头部和文末关联；另一种是调用 world-weaver，把这件事作为固定 Skill 执行。

复盘材料

可复用材料

prompt

YAML + Obsidian 双向链接完整提示词

这段提示词要完整保留，学员回去处理自己的 Markdown 知识库时可以直接复制使用。

我有一批 Markdown 文件需要你帮我处理，让它们更适合作为 AI 知识库来使用。

在文章顶部添加 YAML 头部。

根据我的知识库的实际内容和架构，自己设计最合适的字段。

字段设计的原则：
- 能帮助快速定位这篇文章是什么
- 能说明这篇文章在什么场景下应该被调用
- 能体现它在我的知识体系里的位置和分类
- 字段数量不要太多，够用就好，3-6个为宜

在文章底部添加 Obsidian 双向链接。

读完所有文件后，找出这篇文章和其他文件之间真实存在的逻辑关联。

格式如下：
## 相关文件
- [[文件名]] — 说明这两篇文章之间的关联关系是什么

注意：
- 只链接真正有关联的文件，不要为了链接而链接
- 每条链接后面必须附上一句话，说明两者之间的关系是什么
- 关联关系可以是：互相支撑、互相补充、因果关系、同一主题的不同角度等
skill

world-weaver：把文件堆织成知识网络

当项目里已经有几十篇 Markdown，world-weaver 的价值就出来了。它不是给你做漂亮图谱，而是给每个文件写身份证，再给相关文件建立强连接。

world-weaver

给 Markdown 文件写入 YAML frontmatter，并用 Obsidian 双链织出语义关联网络。

产物
每个文件头部的 YAML + 文末的 ## 关联
什么时候用
项目里已经有一批 Markdown 文件，但它们还是散落文件堆、彼此没有关系时。
前置条件
建议先完成 world-builder，至少要有 CLAUDE.md 和稳定项目结构。
workflow

双链质量自查

双链不是越多越好，链错了会污染知识库。

  1. 只链接真正有关联的文件，不要为了链接而链接。
  2. 每条链接后面写一句关系说明，不要只放文件名。
  3. 如果只是关键词相同，不算强关联。
  4. 如果一个文件被大量无意义引用，它可能变成大哥节点，要回头清理。
  5. 跨业务域链接要更谨慎，必须说清为什么这个跨域有意义。
